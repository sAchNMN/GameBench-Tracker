# 需求与边界

## 一句话定义

GameBench Tracker 是本地单用户的游戏性能记录工具，用于手动记录、管理、对比和导出游戏 Benchmark 结果。

## 项目目标

- 记录游戏、测试场景、配置模板和测试结果。
- 对比同一游戏下两条测试记录。
- 展示基础性能、功耗、温度和能效图表。
- 导出 CSV，兼容中文 Excel。
- 空闲时不持续占用 CPU。
- 关闭后 Java 进程完全退出。

## MVP 范围

MVP 只包含：

- 游戏管理。
- 测试场景管理。
- 配置模板管理。
- 测试记录管理。
- 同一游戏下双记录对比。
- 基础柱状图。
- CSV 导出。
- 输入校验。
- 统一错误处理。

## MVP 非目标

以下内容不得进入 MVP：

- 用户注册、用户登录、多用户、复杂权限。
- JWT、Redis、MySQL、PostgreSQL、Docker、微服务。
- 云同步、在线账号、社区、排行榜。
- AI 分析、AI 推荐配置。
- 自动读取硬件、硬件传感器读取、实时采集 FPS。
- 自动运行游戏、显卡超频控制、驱动控制、游戏进程注入。
- 开机自启、后台常驻、定时任务。
- 多语言、主题系统、移动端、PDF 导出。

## 用户流程

1. 创建游戏。
2. 创建测试场景。
3. 创建或选择配置模板。
4. 填写分辨率、画质、缩放、频率、电压、功耗限制等参数。
5. 使用外部工具完成测试。
6. 录入平均 FPS、1% Low、0.1% Low、温度、功耗等结果。
7. 保存测试记录。
8. 选择同一游戏下两条测试记录。
9. 查看性能、功耗、温度和 FPS/W 差异。
10. 导出 CSV。

## 字段规则

`game` 表不保存版本字段。游戏版本只保存在测试记录中。

`gpu_power_limit_percent` 表示显卡功耗限制调整百分比，不是瓦数。允许值：

- -10 表示降低 10%。
- 0 表示默认限制。
- 10 表示提高 10%。

数据库和接口校验范围为 -100 到 100。

`gpu_avg_power_w` 表示实际平均功耗，单位 W。

`average_fps` 必填，且必须大于 0。

其他 FPS 指标可以为空，也可以为 0，但不能为负数。

## 测试场景规则

新增测试记录时：

- `scene_id` 必填。
- 场景必须存在。
- 场景必须属于所选游戏。
- Service 将场景名称、测试方法、测试时长复制到快照字段。

历史记录中：

- 场景删除后，`scene_id` 可以因 `ON DELETE SET NULL` 变为空。
- 场景快照字段必须继续保留。
- 场景删除不得影响历史记录内容。

## 配置模板规则

新增测试记录时，如果选择配置模板，Service 将模板字段复制到 `benchmark_record`。

保存后：

- 历史测试记录不依赖模板当前值。
- 编辑模板不得改变历史测试记录。
- 删除模板不得改变历史测试记录。
- `config_template_id` 只用于记录来源。

## 历史记录编辑规则

允许编辑历史测试记录，用于修正录入错误。

编辑后必须更新 `updated_at`。

界面必须明确提示用户正在修改历史测试数据。

MVP 不做修改历史审计。

## 时间规范

- 数据库存储 UTC。
- API 使用 ISO 8601。
- 前端按用户本地时区显示。

## 重复提交规则

MVP 不引入 Redis、锁、Token 或复杂幂等机制。

只采用：

- 前端提交期间禁用按钮。
- 数据库唯一约束。
- 标准错误处理。

## 退出模型

MVP 不设计公开的无保护 shutdown HTTP 接口。

后续实现时，采用显式启动器或启动窗口控制 Java 进程生命周期。

关闭软件后，后端 Java 进程必须完全退出。
